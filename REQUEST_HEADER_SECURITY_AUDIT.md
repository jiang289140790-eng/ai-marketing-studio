# Request Header Security Audit

## Scope

Audit target:

- `src/services`
- `src/lib`
- `src/api`
- relevant `supabase/functions` request code

Focus:

- `new Headers`
- `Headers.set`
- `Headers.append`
- `fetch` headers
- axios headers
- Supabase Storage upload headers
- whether user-controlled values enter HTTP headers

User-controlled values checked:

- file name
- Chinese category labels
- Prompt content
- character name
- Workflow name
- LoRA / checkpoint / model names

## Summary

Overall status: **PASS with current safeguards**

No direct path was found where file names, Prompt content, character names, Workflow names, or Chinese labels are written into HTTP headers.

The previous high-risk browser-side `Headers.set()` path has been mitigated by `fetchWithSafeHeaders()` and ASCII header sanitization in:

- `src/services/supabase-client.js`
- `src/services/asset-service.js`

## Detailed Findings

### 1. Supabase global fetch wrapper

File:

- `src/services/supabase-client.js`

Relevant behavior:

- Exports `fetchWithSafeHeaders()`
- Sanitizes header names through `sanitizeHttpHeaderName()`
- Sanitizes header values through `sanitizeHttpHeaderValue()`
- Merges request headers into a plain object instead of constructing unsafe `Headers.set()` calls directly

Risk:

- Low

Assessment:

- This is the main mitigation against browser errors such as:

```text
Failed to execute 'set' on 'Headers': String contains non ISO-8859-1 code point.
```

No user-controlled Prompt/file/category/character/workflow value is intentionally added to headers here.

### 2. Asset upload request

File:

- `src/services/asset-service.js`

Headers used:

```js
{
  apikey: supabaseAnonKey,
  Authorization: `Bearer ${accessToken}`,
  'Content-Type': sanitizeHttpHeaderValue(file.type || 'application/octet-stream') || 'application/octet-stream',
  'cache-control': '3600',
  'x-upsert': 'false'
}
```

User-controlled values:

- `file.name`: used for database display name and extension extraction only
- `metadata.name`: database field only
- `metadata.prompt`: database field only
- `metadata.workflow`: database JSON field only

Risk:

- Low

Assessment:

- File name does not enter `Content-Disposition` or any custom header.
- Storage path uses UUID + sanitized extension.
- `Content-Type` comes from browser file MIME type and is ASCII-sanitized.
- Prompt and workflow metadata are inserted into database body, not headers.

## Remaining Low-Level Risks

1. **Manual future fetch calls**

   If new code uses native `globalThis.fetch()` with user-controlled headers instead of `fetchWithSafeHeaders()`, the same browser error could return.

   Recommendation:

   - In frontend code, use `fetchWithSafeHeaders()` for any manual fetch with headers.

2. **Supabase SDK internals**

   Supabase SDK-generated headers are passed through the configured global fetch wrapper in current client setup.

   Recommendation:

   - Keep `global.fetch = fetchWithSafeHeaders` in Supabase client configuration.

3. **Edge Function secrets**

   Provider API keys and webhook secrets enter server-side headers.

   Recommendation:

   - Keep secrets ASCII-only.
   - Never store user input as a provider API key or webhook secret.

## Final Conclusion

Current request layer is safe against the known non-ASCII header issue.

The earlier error was most likely caused by a header construction path that could attempt to set non-ASCII header values. The current frontend request layer sanitizes headers before fetch, and audited user-controlled values do not enter HTTP headers.
